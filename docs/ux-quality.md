# Проверки качества UX

## Локальный запуск

Из `frontend/admin` выполните `npm ci`, `npx playwright install chromium`, затем `npm run test`, `npm run build-storybook` и `npm run e2e:ux`. Сборки админки и виджета проверяются командой `node .github/scripts/check-bundle-budgets.mjs` из корня репозитория.

Эталонные размеры: телефон 390×844, планшет 768×1024 и компьютер 1440×900. Для каждого критического сценария учитываются светлая и тёмная темы, клавиатура, видимый фокус и `prefers-reduced-motion`.

## Обновление эталонов

Эталон обновляется только вместе с осознанным изменением интерфейса. Linux-эталоны создаются в закреплённом образе `mcr.microsoft.com/playwright:v1.61.1-noble`, который использует CI; Windows-эталоны хранятся отдельно. Перед фиксацией снимков нужно просмотреть различия во всех размерах и обеих темах, запустить axe-проверку и описать изменение в PR. Случайное принятие всех снимков без просмотра недопустимо.

## Бюджеты

- виджет: не более 20 КБ gzip;
- основной пакет админки: не более 180 КБ gzip;
- каждый лениво загружаемый маршрут: не более 60 КБ gzip.

Превышение блокирует CI. Если продуктовая ценность требует пересмотра бюджета, обоснование и измерения должны быть указаны в PR до изменения лимита.

## Качество автомодерации

Детерминированный evaluator проверяется размеченными корпусами `backend/src/test/resources/automod/corpus-ru.jsonl` и `corpus-en.jsonl`. Корпус содержит только синтетические или разрешённые для тестирования тексты без реальных персональных данных. Каждая запись имеет стабильный `id`, язык, текст, ожидаемый класс `SAFE`, `REVIEW` или `SPAM` и безопасные tags. Полный текст ошибочного примера не выводится в CI-лог: отчёт показывает только ID, tags и confusion matrix.

На каждый язык требуется не менее 180 примеров: минимум по 60 для каждого класса, не менее 20 adversarial-примеров с обфускацией/homoglyph/leetspeak и не менее 20 допустимых сообщений со ссылками, числами или контактными словами для контроля false positive. Тест отдельно блокирует дубли ID/текстов, пустые классы, управляющие символы и нарушение минимального распределения.

Для default-политики `BALANCED` все пороги применяются отдельно к RU и EN:

- precision автоматического `APPROVE` — не ниже 98%;
- precision решения `SPAM` — не ниже 97%;
- доля настоящего `SPAM`, ошибочно получившего `APPROVE`, — не выше 5%;
- доля `SAFE`, ошибочно получившего `SPAM`, — не выше 2%;
- recall класса `REVIEW` — не ниже 75%;
- macro-F1 по трём классам — не ниже 0,85;
- разница RU/EN macro-F1 — не больше 0,05.

Дополнительные safety-гейты: для `OPEN` precision `SPAM` не ниже 99% и `SAFE -> SPAM` равен нулю; для `STRICT` precision `APPROVE` не ниже 99% и `SPAM -> APPROVE` равен нулю. На всём корпусе решение должно быть монотонным в порядке `OPEN <= BALANCED <= STRICT`, где `APPROVE < REVIEW < SPAM`. Boundary-тесты отдельно проверяют `reviewThreshold - 1`, `reviewThreshold`, `spamThreshold - 1` и `spamThreshold`.

Corpus-тест входит в обычный backend `test` и потому блокирует required check. При изменении правил PR обязан приложить новый отчёт по обоим языкам; снижать пороги или подбирать корпус под реализацию без объяснения и ревью нельзя.

## E2E автомодерации

Production-like Playwright-сценарий проверяет полный lifecycle политики:

1. Создание черновика `BALANCED`, моделирование безопасного, неоднозначного и спамного текста без создания комментария и изменения очереди.
2. Публикацию `LIVE`: безопасный комментарий становится публичным, неоднозначный получает `PENDING`, спам — `SPAM` и не попадает в public list.
3. `SHADOW`: score/decision/signals сохраняются, но `PRE_MODERATION` оставляет `PENDING`, а `POST_MODERATION` и `DISABLED` — `APPROVED`.
4. `OPEN` и `BALANCED` автоматически одобряют чистый комментарий при `PRE_MODERATION`; `STRICT` с `FOLLOW_SITE_MODE` оставляет его в `PENDING`.
5. Публикацию `CUSTOM`, границы ссылок/стоп-слов и откат старой версии с новым номером без изменения истории.
6. Конфликт двух вкладок: первая сохраняет revision, вторая получает `409`, сохраняет несохранённый ввод и предлагает загрузить актуальную версию.
7. Идемпотентные `FALSE_POSITIVE`/`FALSE_NEGATIVE`, удаление отметки и owner isolation.
8. Отсутствие запросов к внешним AI/provider hosts при моделировании, публикации и проверке комментария.

Таблица версий, результаты моделирования, conflict dialog и feedback-действия покрываются клавиатурой, axe и размерами 390×844, 768×1024 и 1440×900. `prefers-reduced-motion` не скрывает состояние публикации, ошибки или смену активной версии.

## Изоляция виджета

Виджет живёт в Shadow DOM: стили страницы не должны попадать внутрь, а стили виджета — наружу. Сценарии проверяют контейнеры от 320 до 1440 пикселей, загрузку, пустое состояние, ошибку, авторизацию, ответы, реакции и ожидание модерации.

Редактор оформления использует реальный код виджета в sandboxed iframe. Родитель передаёт только типизированную конфигурацию V2 через `postMessage`; тестовый транспорт возвращает фиксированные комментарии и не обращается к production API. Публичный контракт принимает только перечислимые дизайн-параметры, цвет `#RRGGBB` и короткие подписи без HTML и управляющих символов.

Адаптивность виджета определяется шириной его контейнера, а не viewport внешней страницы. Container queries покрывают 320–1440 пикселей; форма входа раскрывается после явного намерения, профильные и privacy-действия находятся в доступном меню, длинные ветки догружаются страницами. Создание, редактирование, удаление и реакции показываются оптимистично и возвращаются к подтверждённому состоянию при ошибке.

## Центр уведомлений

Центр уведомлений использует постоянный API как источник истины, а WebSocket — для доставки новых элементов без перезагрузки. Идентификатор постоянного уведомления участвует в дедупликации, поэтому reconnect и повторная доставка не создают две строки. Непрочитанное состояние сохраняется между вкладками и сессиями; открытие панели само по себе ничего не отмечает прочитанным.

Кнопка уведомлений остаётся на месте при открытии portal-панели. Панель получает фокус, закрывается по `Escape` и клику снаружи, возвращает фокус на кнопку и показывает явные действия «Отметить всё» и переход к конкретному комментарию. Оптимистичная отметка прочтения откатывается при ошибке API, а потеря realtime-соединения не скрывает уже сохранённые уведомления.

## Создание и рабочее пространство сайта

Мастер создания показывает один смысловой шаг за раз, проверяет текущий шаг перед переходом и переводит фокус на первое неверное поле. Черновик хранится локально семь дней в ключе конкретного владельца, не переносится между аккаунтами и удаляется после успешного создания. Недоступность локального хранилища не блокирует мастер. Предварительный просмотр запускает реальный widget bundle в sandboxed iframe с тестовым транспортом.

Страница сайта разделена на обзор, установку, оформление, модерацию, origins и опасную зону. Аналитика не преграждает путь к настройкам. Каждый раздел сохраняется независимо; при уходе с несохранёнными изменениями интерфейс предупреждает пользователя. Установка показывает контрольный список и различает отсутствие подключения, здоровое, устаревшее и отклонённое состояния без серверной загрузки страницы клиента.
