# Модель данных

## Пользователи и роли

CloudComment хранит пользователей приложения в таблице `app_users`.

`app_users` совместима с будущими задачами регистрации, входа и авторизации:

- `id` — сгенерированный UUID, первичный ключ.
- `email` — уникальный адрес, размер поля подходит для RFC-совместимых email.
- `password_hash` — хеш пароля, не исходный пароль.
- `display_name` — необязательное отображаемое имя.
- `is_enabled` — позволяет отключить доступ; flow удаления аккаунта также переводит пользователя в disabled-состояние.
- `deleted_at` — timestamp подтвержденного удаления аккаунта; активная авторизация учитывает только пользователей с `deleted_at is null`.
- `created_at` и `updated_at` поддерживают аудит и будущие сценарии управления профилем.

Роли хранятся в справочной таблице `roles` и назначаются пользователям через `user_roles`.

Роли MVP:

- `OWNER` — владелец сайта с административным доступом к своим ресурсам.
- `COMMENTER` — авторизованный посетитель, который может публиковать комментарии.
- `MODERATOR` — роль, зарезервированная для будущих сценариев модерации.

`user_roles` хранит связь many-to-many между пользователями и ролями. Это позволит будущей авторизации поддерживать пользователей с несколькими наборами прав без изменения основной таблицы пользователей.

## Авторизационные сессии

После успешного входа CloudComment создает запись в таблице `auth_sessions`.

- `id` — сгенерированный UUID сессии.
- `user_id` — ссылка на пользователя из `app_users`; при подтвержденном удалении аккаунта активные сессии пользователя отзываются.
- `token_hash` — SHA-256 хеш выданного bearer-токена. Исходный токен возвращается клиенту один раз и не хранится в базе.
- `created_at`, `expires_at` и `revoked_at` задают срок действия; выход пользователя заполняет `revoked_at` у текущей сессии.

`auth_sessions` хранит только серверную сторону авторизационного токена. Endpoint текущего пользователя и общий защитный слой backend API используют эту таблицу для проверки bearer-токена.

## Удаление аккаунта

Запросы на удаление аккаунта хранятся в `account_deletion_requests`.

- `user_id` — пользователь, который запросил удаление.
- `token_hash` — SHA-256 хеш одноразового confirmation token; исходный token не хранится в базе.
- `created_at` и `expires_at` задают срок действия confirmation token.
- `confirmed_at` фиксирует успешное подтверждение удаления.
- `cancelled_at` закрывает устаревшие или замененные pending-заявки.

Для пользователя допускается только одна pending-заявка на удаление. Повторный запрос с активной заявкой ротирует confirmation token и продлевает срок действия. Если предыдущая pending-заявка уже истекла, она помечается `cancelled_at`, после чего создается новая заявка.

После подтверждения удаления аккаунт не удаляется физически из `app_users`: email заменяется псевдонимизированным значением `deleted-{userId}@deleted.invalid`, пароль становится непригодным для входа, `display_name` очищается, `is_enabled=false`, `deleted_at` заполняется, а все активные `auth_sessions` отзываются.

Связанные данные обрабатываются транзакционно:

- сайты, которыми владел удаляемый пользователь, удаляются из `sites`; связанные `site_allowed_origins`, `pages`, `comments` и `moderation_actions` удаляются каскадно;
- комментарии, которые пользователь оставил на чужих сайтах, остаются только как обезличенный след обсуждения: `author_user_id` и `author_email` очищаются, `author_name` заменяется на `Deleted user`, а текст заменяется на `Comment deleted by user`;
- строки `moderation_actions`, где пользователь был модератором, очищают `moderator_id`.

`GET /api/account/personal-data` возвращает текущему пользователю snapshot собственных данных: профиль, роли, согласия, сводку сессий, счетчики owned/authored/moderation ресурсов и последнюю заявку на удаление. Endpoint не раскрывает данные других пользователей или содержимое чужих ресурсов.

Встраиваемый виджет использует site-scoped aliases под `/api/public/sites/{siteId}/account/*` для экспорта персональных данных и запроса удаления аккаунта. Эти маршруты возвращают те же данные текущего пользователя, но дополнительно проходят domain policy сайта, чтобы external-origin CORS не открывал глобальные `/api/account/*`.

## Privacy audit и retention

Минимальный privacy-аудит хранится в `privacy_events`.

- `user_id` — ссылка на пользователя, если событие относится к субъекту данных.
- `event_type` — стабильный тип события: согласие, заявка на удаление, подтверждение, удаление/анонимизация, экспорт данных или retention cleanup.
- `metadata` — JSONB только с безопасной технической метадатой; raw password, bearer/session token и confirmation token не сохраняются.
- `created_at` — время события.

Retention cleanup закрывает истекшие pending-заявки на удаление, удаляет старые confirmed/cancelled deletion requests после retention window и чистит старые revoked/expired sessions. Значения по умолчанию задаются через `cloud-comment.privacy.session-retention-days`, `cloud-comment.privacy.deletion-request-retention-days` и `cloud-comment.privacy.retention-cleanup-cron`.

## Владение ресурсами

Владение административными ресурсами определяется через `sites.owner_id`.

Admin API для MVP управляет проектами как сайтами: создание проекта/сайта создаёт строку в `sites`, а отдельная таблица `projects` не используется. `sites.public_key` генерируется backend-ом и остаётся уникальным стабильным идентификатором сайта; разрешённые origin-ы виджета хранятся в `site_allowed_origins`.

Настройки внешнего вида виджета хранятся рядом с сайтом в `sites`: `widget_theme` (`AUTO`, `LIGHT`, `DARK`), `widget_accent_color` (`#RRGGBB`) и `widget_corner_radius` (`SMALL`, `MEDIUM`, `LARGE`). Public widget config возвращает эти значения, чтобы один и тот же bundle мог отрисовываться под стиль конкретного сайта без отдельной таблицы конфигурации.

Настройки автомодерации тоже site-scoped и лежат в `sites`: `automod_enabled`, `automod_strictness` (`OFF`, `RELAXED`, `BALANCED`, `STRICT`), `automod_blocked_words`, `automod_hold_links`, `automod_block_links` и `automod_max_links`. Такой уровень выбран намеренно: правила модерации принадлежат владельцу конкретного сайта и могут отличаться между проектами одного пользователя.

Удаление сайта через Admin API удаляет строку `sites`; связанные `site_allowed_origins`, `pages`, `comments` и `moderation_actions` удаляются каскадно по внешним ключам.

Ресурсы, привязанные к сайту, наследуют владельца по связям:

- `site_allowed_origins.site_id -> sites.id`
- `pages.site_id -> sites.id`
- `comments.page_id -> pages.id -> sites.id`
- `moderation_actions.comment_id -> comments.id -> pages.id -> sites.id`

Проверка владения не использует автора комментария или модератора действия как владельца административного ресурса. Если ресурс отсутствует или принадлежит другому пользователю, API возвращает единый `404 NOT_FOUND` с сообщением `Resource not found`, чтобы не раскрывать существование чужих UUID.

## Страницы, комментарии и public widget policy

Public Widget API привязывает обсуждение к паре `sites.id` и `pages.url`. Если visitor впервые оставляет комментарий на странице, backend создает строку в `pages`, затем сохраняет комментарий в `comments` с `page_id`.

Ответы на комментарии используют `comments.parent_id`. В MVP виджет создает ответы только к корневым approved-комментариям и показывает их одним уровнем вложенности; неподходящий parent comment маскируется как `404 NOT_FOUND`.

Публичный доступ виджета ограничен domain policy:

- `Origin` запроса должен совпадать с одной из записей `site_allowed_origins.origin` для активного сайта.
- `pageUrl` должен принадлежать тому же origin, чтобы один разрешенный сайт не мог читать или создавать обсуждения для чужого URL.
- Неразрешенный origin, inactive/missing site или parent comment не из той же страницы маскируются как `404 NOT_FOUND`.

Создание комментария в MVP требует bearer-сессию пользователя. `author_user_id`, `author_email` и `author_name` заполняются из текущего пользователя, чтобы будущая модерация могла показывать автора.

Создание и редактирование комментария проходят автомодерацию сайта. Базовый режим модерации задает обычный начальный статус, а автомодерация может усилить решение до `PENDING` или `SPAM` по стоп-словам, ссылкам, токсичным/спамным маркерам и выбранной строгости.

Автор комментария может редактировать и удалять только свой комментарий через public widget API. Редактирование обновляет `content`, `status`, `moderation_reason`, `updated_at` и `edited_at`, затем комментарий снова проходит правила видимости. Удаление автором является soft-delete: `deleted_at` и `deleted_by_author` заполняются, текст заменяется безопасной служебной строкой, статус переводится в `HIDDEN`, а реакции к комментарию удаляются. Soft-deleted comments не возвращаются в public list и moderation queue.

Публичный список комментариев виджета возвращает только `APPROVED`. Комментарии `PENDING`, `REJECTED`, `HIDDEN` и `SPAM` остаются в базе для административной модерации, но не видны внешнему сайту.

Реакции пользователей хранятся в `comment_reactions`. На один комментарий у одного пользователя может быть только одна активная реакция; повторная установка другого типа заменяет старую, а `null` в public reaction endpoint удаляет реакцию. Реакции привязаны к `comments.id` и `app_users.id`, каскадно удаляются вместе с комментариями и возвращаются в public list как агрегированные счетчики с флагом `reactedByCurrentUser` для текущей bearer-сессии. При подтвержденном удалении аккаунта реакции пользователя удаляются, а экспорт персональных данных показывает их количество в `resources.commentReactions`.

### Аналитика владельца сайта

Owner analytics не добавляет новых таблиц: read-model строится поверх `sites`, `pages`, `comments` и `comment_reactions`.

- Доступ всегда ограничен через `sites.owner_id`; при `siteId` backend дополнительно проверяет владение сайтом и маскирует чужие UUID как `404 NOT_FOUND`.
- Счетчики комментариев, ответов, статусов, топ страниц и активных авторов используют `comments.created_at` и исключают soft-deleted comments (`deleted_at is null`).
- Счетчики реакций используют `comment_reactions.created_at`, но также исключают реакции к soft-deleted comments.
- Периоды `7d`, `30d` и `90d` агрегируются по дням, `all` агрегируется по месяцам.

### Объяснимая автомодерация

Автомодерация работает на прозрачных правилах: создание и редактирование комментария нормализуют текст, проверяют стоп-слова владельца, ссылки, spam/toxicity-маркеры, обфускацию, повторяющиеся символы, капс и подозрительные контакты. Если решение требует внимания владельца, короткая причина сохраняется в `comments.moderation_reason`.

Владелец сайта может выполнить dry-run через `POST /api/sites/{siteId}/automoderation/check`. Endpoint owner-scoped, возвращает `status`, `score`, nullable `reason` и отдельные `signals`, но не пишет комментарии в базу. Чужие или отсутствующие сайты маскируются как `404 NOT_FOUND`.

`SPAM`-комментарии остаются в базе для модерации и не видны в публичном списке виджета. Пользователь виджета получает только нейтральное сообщение "отправлено на проверку"; внутренние spam-сигналы показываются только владельцу в админской модерации.
