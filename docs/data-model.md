# Модель данных

## Пользователи и роли

CloudComment хранит пользователей приложения в таблице `app_users`.

`app_users` совместима с будущими задачами регистрации, входа и авторизации:

- `id` — сгенерированный UUID, первичный ключ.
- `email` — уникальный адрес, размер поля подходит для RFC-совместимых email.
- `password_hash` — хеш пароля, не исходный пароль.
- `display_name` — необязательное отображаемое имя.
- `is_enabled` — позволяет отключить доступ без удаления аккаунта.
- `created_at` и `updated_at` поддерживают аудит и будущие сценарии управления профилем.

Роли хранятся в справочной таблице `roles` и назначаются пользователям через `user_roles`.

Роли MVP:

- `OWNER` — владелец сайта с административным доступом к своим ресурсам.
- `COMMENTER` — авторизованный посетитель, который может публиковать комментарии.
- `MODERATOR` — роль, зарезервированная для будущих сценариев модерации.

`user_roles` хранит связь many-to-many между пользователями и ролями. Это позволит будущей авторизации поддерживать пользователей с несколькими наборами прав без изменения основной таблицы пользователей.

## Авторизационные сессии

После успешного входа CloudComment создает запись в таблице `auth_sessions`.

- `id` — сгенерированный UUID сессии.
- `user_id` — ссылка на пользователя из `app_users`; сессии удаляются вместе с пользователем.
- `token_hash` — SHA-256 хеш выданного bearer-токена. Исходный токен возвращается клиенту один раз и не хранится в базе.
- `created_at`, `expires_at` и `revoked_at` задают срок действия; выход пользователя заполняет `revoked_at` у текущей сессии.

`auth_sessions` хранит только серверную сторону авторизационного токена. Endpoint текущего пользователя и общий защитный слой backend API используют эту таблицу для проверки bearer-токена.

## Владение ресурсами

Владение административными ресурсами определяется через `sites.owner_id`.

Admin API для MVP управляет проектами как сайтами: создание проекта/сайта создаёт строку в `sites`, а отдельная таблица `projects` не используется. `sites.public_key` генерируется backend-ом и остаётся уникальным стабильным идентификатором сайта; разрешённые origin-ы виджета хранятся в `site_allowed_origins`.

Ресурсы, привязанные к сайту, наследуют владельца по связям:

- `site_allowed_origins.site_id -> sites.id`
- `pages.site_id -> sites.id`
- `comments.page_id -> pages.id -> sites.id`
- `moderation_actions.comment_id -> comments.id -> pages.id -> sites.id`

Проверка владения не использует автора комментария или модератора действия как владельца административного ресурса. Если ресурс отсутствует или принадлежит другому пользователю, API возвращает единый `404 NOT_FOUND` с сообщением `Resource not found`, чтобы не раскрывать существование чужих UUID.
