# Модель данных

## Пользователи и роли

CloudComment хранит пользователей приложения в таблице `app_users`.

`app_users` совместима с будущими задачами регистрации, входа и авторизации:

- `id` — сгенерированный UUID, первичный ключ.
- `email` — уникальный адрес, размер поля подходит для RFC-совместимых email.
- `password_hash` — хеш пароля, не исходный пароль.
- `display_name` — необязательное отображаемое имя.
- `is_enabled` — позволяет отключить доступ; flow удаления аккаунта также переводит пользователя в disabled-состояние.
- `deleted_at` — timestamp подтвержденного удаления аккаунта; активная авторизация учитывает только пользователей с `deleted_at is null`.
- `created_at` и `updated_at` поддерживают аудит и будущие сценарии управления профилем.

Роли хранятся в справочной таблице `roles` и назначаются пользователям через `user_roles`.

Роли MVP:

- `OWNER` — владелец сайта с административным доступом к своим ресурсам.
- `COMMENTER` — авторизованный посетитель, который может публиковать комментарии.
- `MODERATOR` — роль, зарезервированная для будущих сценариев модерации.

`user_roles` хранит связь many-to-many между пользователями и ролями. Это позволит будущей авторизации поддерживать пользователей с несколькими наборами прав без изменения основной таблицы пользователей.

## Авторизационные сессии

После успешного входа CloudComment создает запись в таблице `auth_sessions`.

- `id` — сгенерированный UUID сессии.
- `user_id` — ссылка на пользователя из `app_users`; при подтвержденном удалении аккаунта активные сессии пользователя отзываются.
- `token_hash` — SHA-256 хеш raw session token. Исходный token не хранится в базе; для admin он передаётся только в HttpOnly host-only cookie, а для public widget API пока остаётся отдельным bearer до изоляции сессий в #168.
- `audience` — обязательное назначение `ADMIN`, `WIDGET` или rollback-совместимое `LEGACY`. Новый backend создаёт только первые два значения и всегда сверяет ожидаемый audience; `LEGACY` не аутентифицирует ни один маршрут.
- `created_at`, `expires_at` и `revoked_at` задают срок действия; выход пользователя заполняет `revoked_at` у текущей сессии.

`auth_sessions` хранит только серверную сторону авторизационного токена. Admin API проверяет raw token только из HttpOnly cookie и требует audience `ADMIN`; bearer-заголовок для admin игнорируется. Public routes `/api/public/sites/**` не используют admin cookie и требуют audience `WIDGET` для авторизованных действий. Миграция audience отзывает все старые неразмеченные сессии, а `LEGACY` default оставляет схему совместимой с автоматическим откатом на предыдущий backend без признания таких сессий новым кодом. CSRF secret хранится в отдельной HttpOnly cookie и памяти вкладки, не сохраняется в базе и проверяется по схеме cookie/header для всех изменяющих admin-запросов.

## Удаление аккаунта

Запросы на удаление аккаунта хранятся в `account_deletion_requests`.

- `user_id` — пользователь, который запросил удаление.
- `token_hash` — SHA-256 хеш одноразового confirmation token; исходный token не хранится в базе.
- `created_at` и `expires_at` задают срок действия confirmation token.
- `confirmed_at` фиксирует успешное подтверждение удаления.
- `cancelled_at` закрывает устаревшие или замененные pending-заявки.

Для пользователя допускается только одна pending-заявка на удаление. Повторный запрос с активной заявкой ротирует confirmation token и продлевает срок действия. Если предыдущая pending-заявка уже истекла, она помечается `cancelled_at`, после чего создается новая заявка.

После подтверждения удаления аккаунт не удаляется физически из `app_users`: email заменяется псевдонимизированным значением `deleted-{userId}@deleted.invalid`, пароль становится непригодным для входа, `display_name` очищается, `is_enabled=false`, `deleted_at` заполняется, а все активные `auth_sessions` отзываются.

Связанные данные обрабатываются транзакционно:

- сайты, которыми владел удаляемый пользователь, удаляются из `sites`; связанные `site_allowed_origins`, `pages`, `comments` и `moderation_actions` удаляются каскадно;
- комментарии, которые пользователь оставил на чужих сайтах, остаются только как обезличенный след обсуждения: `author_user_id` и `author_email` очищаются, `author_name` заменяется на `Deleted user`, а текст заменяется на `Comment deleted by user`;
- строки `moderation_actions`, где пользователь был модератором, очищают `moderator_id`.

`GET /api/account/personal-data` возвращает текущему пользователю snapshot собственных данных: профиль, роли, согласия, сводку сессий, счетчики owned/authored/moderation ресурсов и последнюю заявку на удаление. Endpoint не раскрывает данные других пользователей или содержимое чужих ресурсов.

Встраиваемый виджет использует site-scoped aliases под `/api/public/sites/{siteId}/account/*` для экспорта персональных данных и запроса удаления аккаунта. Эти маршруты возвращают те же данные текущего пользователя, но дополнительно проходят domain policy сайта, чтобы external-origin CORS не открывал глобальные `/api/account/*`.

## Privacy audit и retention

Минимальный privacy-аудит хранится в `privacy_events`.

- `user_id` — ссылка на пользователя, если событие относится к субъекту данных.
- `event_type` — стабильный тип события: согласие, заявка на удаление, подтверждение, удаление/анонимизация, экспорт данных или retention cleanup.
- `metadata` — JSONB только с безопасной технической метадатой; raw password, bearer/session token и confirmation token не сохраняются.
- `created_at` — время события.

Retention cleanup закрывает истекшие pending-заявки на удаление, удаляет старые confirmed/cancelled deletion requests после retention window и чистит старые revoked/expired sessions. Значения по умолчанию задаются через `cloud-comment.privacy.session-retention-days`, `cloud-comment.privacy.deletion-request-retention-days` и `cloud-comment.privacy.retention-cleanup-cron`.

## Владение ресурсами

Владение административными ресурсами определяется через `sites.owner_id`.

Admin API для MVP управляет проектами как сайтами: создание проекта/сайта создаёт строку в `sites`, а отдельная таблица `projects` не используется. `sites.public_key` генерируется backend-ом и остаётся уникальным стабильным идентификатором сайта; разрешённые origin-ы виджета хранятся в `site_allowed_origins`.

Настройки внешнего вида виджета хранятся рядом с сайтом в `sites`. Миграция V10 добавляет версионированные `widget_style_version` и `widget_style_config JSONB`: конфигурация управляет темой, плотностью, читаемой шириной, выравниванием, типографикой, заголовком, положением редактора, сортировкой, реакциями, аватарами, глубиной, локалью и безопасными текстовыми подписями. Произвольные CSS/HTML и юридические тексты в контракт не входят. Legacy-поля `widget_theme`, `widget_accent_color` и `widget_corner_radius` сохраняются; новый backend пишет оба представления, а триггер V10 синхронизирует JSONB при записи старым релизом. Поэтому предыдущий release можно безопасно вернуть после добавочной миграции.

Предварительный просмотр в админке запускает тот же bundle виджета в sandboxed iframe. Конфигурация передаётся через `postMessage`, а запросы перехватывает локальный тестовый транспорт: preview не читает боевые комментарии и не загружает страницы клиента.

Legacy-настройки автомодерации остаются site-scoped в `sites`: `automod_enabled`, `automod_strictness` (`OFF`, `RELAXED`, `BALANCED`, `STRICT`), `automod_blocked_words`, `automod_hold_links`, `automod_block_links` и `automod_max_links`. После миграции V14 они служат совместимым представлением активной политики для предыдущих клиентов и безопасного отката приложения. Источником истины для новых операций являются версионируемые политики, описанные ниже.

Удаление сайта через Admin API удаляет строку `sites`; связанные `site_allowed_origins`, `pages`, `comments`, `moderation_actions`, версии политик и обратная связь по автомодерации удаляются каскадно по внешним ключам.

Ресурсы, привязанные к сайту, наследуют владельца по связям:

- `site_allowed_origins.site_id -> sites.id`
- `pages.site_id -> sites.id`
- `comments.page_id -> pages.id -> sites.id`
- `moderation_actions.comment_id -> comments.id -> pages.id -> sites.id`
- `automod_policy_versions.site_id -> sites.id`
- `site_automod_policy_state.site_id -> sites.id`
- `automod_policy_feedback.comment_id -> comments.id -> pages.id -> sites.id`

Проверка владения не использует автора комментария или модератора действия как владельца административного ресурса. Если ресурс отсутствует или принадлежит другому пользователю, API возвращает единый `404 NOT_FOUND` с сообщением `Resource not found`, чтобы не раскрывать существование чужих UUID.

## Страницы, комментарии и public widget policy

Public Widget API привязывает обсуждение к паре `sites.id` и `pages.url`. Если visitor впервые оставляет комментарий на странице, backend создает строку в `pages`, затем сохраняет комментарий в `comments` с `page_id`.

`pages.url` хранит канонический URL страницы без fragment, tracking-параметров и чувствительных query-параметров. Backend является источником истины и применяет правила при каждом public list/create запросе, включая запросы старых клиентов. Имена параметров сравниваются без учёта регистра после одного percent-decode: удаляются `utm_*`, `fbclid`, `gclid`, `dclid`, `msclkid`, `yclid`, `twclid`, `ttclid`, `igshid`, `li_fat_id`, `_ga`, `_gl`, `mc_cid`, `mc_eid`, `gbraid`, `wbraid`, `srsltid`, `gad_source`, `gad_campaignid`, а также `access_token`, `id_token`, `refresh_token`, `auth_token`, `authorization`, `api_key`, `apikey`, `jwt`, `password`, `session`, `session_id`, `sessionid`, `token`, `client_secret`, `secret`, `signature`, `sig`, `otp`, `jsessionid`, `phpsessid`, любой префикс `x-amz-`/`x-goog-` и любой суффикс `_token`. Остальные параметры сохраняются с исходным raw encoding, порядком, повторами и пустыми `&`-сегментами, поэтому функциональные query определяют отдельные обсуждения. `code`, `ticket` и `sid` functional-by-default и также сохраняются: OAuth/SSO-интеграция, использующая их как credentials, обязана передавать чистый canonical `pageUrl`/`data-page-url` без секрета. Malformed `%` экранируется как `%25`, не затрагивая корректные percent escapes. Default-порты `https:443` и `http:80` не входят в канонический origin; нестандартные явные порты сохраняются.

Ответы на комментарии используют `comments.parent_id`. Виджет создаёт ответы только к корневым approved-комментариям и показывает их одним уровнем вложенности; неподходящий parent comment маскируется как `404 NOT_FOUND`. Публичный список совместимо принимает необязательный `replyLimit`: без него старые клиенты получают все ответы, с ним каждая ветка содержит первые N ответов и точный `replyCount`. Остальные ответы загружаются страницами через `GET /api/public/sites/{siteId}/comments/{commentId}/replies`; чужая или недоступная ветка возвращает пустую страницу и не раскрывает существование комментария.

Публичный доступ виджета ограничен domain policy:

- `Origin` запроса должен совпадать с одной из записей `site_allowed_origins.origin` для активного сайта.
- `pageUrl` должен принадлежать тому же origin, чтобы один разрешенный сайт не мог читать или создавать обсуждения для чужого URL.
- Неразрешенный origin, inactive/missing site или parent comment не из той же страницы маскируются как `404 NOT_FOUND`.

Создание комментария в MVP требует bearer-сессию пользователя. `author_user_id`, `author_email` и `author_name` заполняются из текущего пользователя, чтобы модерация и privacy lifecycle могли идентифицировать автора. `author_email` остаётся внутренним owner/self-only полем и никогда не сериализуется в Public Widget API. Публичная проекция содержит только `id` и безопасный `displayName`; отсутствующее или email-подобное имя заменяется на `Участник` на API-границе без удаления внутреннего контакта.

Создание и редактирование комментария проходят автомодерацию сайта. Базовый режим модерации задает обычный начальный статус, а автомодерация может усилить решение до `PENDING` или `SPAM` по стоп-словам, ссылкам, токсичным/спамным маркерам и выбранной строгости.

Автор комментария может редактировать и удалять только свой комментарий через public widget API. Редактирование обновляет `content`, `status`, `moderation_reason`, `updated_at` и `edited_at`, затем комментарий снова проходит правила видимости. Удаление автором является soft-delete: `deleted_at` и `deleted_by_author` заполняются, текст заменяется безопасной служебной строкой, статус переводится в `HIDDEN`, а реакции к комментарию удаляются. Soft-deleted comments не возвращаются в public list и moderation queue.

Публичный список комментариев виджета возвращает только `APPROVED`. `comments.is_pinned` доступен только для корневых approved-комментариев: такие ветки всегда идут первыми, а внутри pinned/non-pinned групп применяется `PINNED_FIRST`, `NEWEST`, `OLDEST` или `TOP_REACTIONS`. Для `TOP_REACTIONS` суммируются реакции корня и его approved-ответов. При уходе закреплённого комментария из `APPROVED` флаг автоматически снимается. Комментарии `PENDING`, `REJECTED`, `HIDDEN` и `SPAM` остаются в базе для административной модерации, но не видны внешнему сайту.

`comments.is_favorite` — внутренний owner-scoped флаг очереди модерации. Он фильтруется и изменяется только через Admin API и не включается в public widget response.

Миграция V11 расширяет `moderation_actions` полями `operation_id` и `reverts_action_id`. Пара `(comment_id, operation_id)` обеспечивает идемпотентность массового действия, а уникальная ненулевая ссылка `reverts_action_id` запрещает повторно отменить одно действие. Отмена записывается отдельной строкой с типом `UNDO`; исходная строка остаётся неизменяемой, поэтому история решений сохраняется. Групповая отмена находит исходные действия по `operation_id` только через сайты текущего владельца и выполняет каждый элемент в отдельной транзакции; созданные `UNDO` получают общий новый `operation_id`. Все проверки владения выполняются через комментарий и сайт, а частичный bulk-ответ одинаково маскирует отсутствующие и чужие UUID.

Реакции пользователей хранятся в `comment_reactions`. На один комментарий у одного пользователя может быть только одна активная реакция; повторная установка другого типа заменяет старую, а `null` в public reaction endpoint удаляет реакцию. Реакции привязаны к `comments.id` и `app_users.id`, каскадно удаляются вместе с комментариями и возвращаются в public list как агрегированные счетчики с флагом `reactedByCurrentUser` для текущей bearer-сессии. При подтвержденном удалении аккаунта реакции пользователя удаляются, а экспорт персональных данных показывает их количество в `resources.commentReactions`.

### Аналитика владельца сайта

Аналитика владельца строит модель чтения поверх `sites`, `pages`, `comments`, `comment_reactions`, `moderation_actions` и неизменяемых событий решений автомодерации из V15.

- Доступ всегда ограничен через `sites.owner_id`; при `siteId` backend дополнительно проверяет владение сайтом и маскирует чужие UUID как `404 NOT_FOUND`.
- Счётчики комментариев, ответов, статусов, популярных страниц и активных авторов используют `comments.created_at` и исключают мягко удалённые комментарии (`deleted_at is null`).
- Счётчики реакций используют `comment_reactions.created_at`, но также исключают реакции к мягко удалённым комментариям.
- Все диапазоны полуоткрытые `[from, to)`, а календарные границы и интервалы вычисляются в проверенном часовом поясе IANA владельца. Периоды `7d` и `30d` агрегируются по дням, `90d` — по ISO-неделям, `all` — по месяцам; отсутствующие интервалы заполняются нулями.
- Текущая нагрузка содержит очередь `PENDING + SPAM` и время самого старого ожидающего комментария. Ручные решения и `UNDO` считаются по неизменяемым строкам `moderation_actions`; автоматические решения — только по событиям `LIVE`, поэтому `SHADOW` не выдаётся за сокращение ручной работы.
- Для конечных диапазонов предыдущий период сдвигается на такое же число локальных календарных дней. Процентное изменение не вычисляется при нулевой базе и возвращается как `null`. Для `all` сравнение равно `null`.
- `moderationDistribution` заменяет некорректное название `moderationFunnel`; старое поле остаётся идентичным псевдонимом на один совместимый релиз.

### Объяснимая автомодерация

Автомодерация работает на прозрачных правилах: создание и редактирование комментария нормализуют текст, проверяют стоп-слова владельца, ссылки, spam/toxicity-маркеры, обфускацию, повторяющиеся символы, капс и подозрительные контакты. Внешний AI-провайдер не используется, содержимое комментария не отправляется за пределы CloudComment.

Миграция `V14__automod_policy_versions.sql` добавляет версионируемые политики, состояние политики сайта, snapshot решений и обратную связь владельца.

`automod_policy_versions` хранит:

- `site_id` и последовательный `version_number`, уникальный внутри сайта;
- `lifecycle` `DRAFT` или `PUBLISHED` (в API — `state`), режим `SHADOW` или `LIVE`, preset `OPEN`, `BALANCED`, `STRICT` или `CUSTOM`;
- пороги review/spam, действие для чистого комментария и безопасную конфигурацию ссылок и стоп-слов;
- `revision` для optimistic concurrency, автора и время создания/изменения/публикации;
- ссылку на исходную версию для публикации или отката.

`site_automod_policy_state` содержит по одной строке на сайт: `active_policy_version_id`, эффективные `enabled` и `execution_mode`, последний опубликованный номер, fingerprint legacy-настроек и время изменения. Единственный черновик определяется partial unique index по `site_id` среди строк с `lifecycle = 'DRAFT'`; отдельного draft pointer нет. Черновик можно обновлять и удалять до публикации, каждое изменение атомарно увеличивает `revision`, а удаление также требует ожидаемую revision. Опубликованные версии защищены trigger-ом от изменения и не удаляются поштучно. Публикация атомарно переводит согласованный `DRAFT` в `PUBLISHED`, назначает следующий номер и меняет active pointer. Откат создаёт новую published-версию из выбранной старой: выбранная и ранее активная строки остаются неизменными. Клиент передаёт ожидаемую revision и, при публикации, ожидаемый идентификатор активной версии. Конкурирующая запись с устаревшими ожиданиями получает `409 CONFLICT`, поэтому две вкладки не могут молча перезаписать или удалить более новый черновик.

Fingerprint обеспечивает двустороннюю совместимость с предыдущим release. Новый backend синхронно обновляет legacy-колонки и fingerprint при публикации. Если старый backend позднее изменил `sites.automod_*`, новый release обнаруживает расхождение и под блокировкой создаёт новую опубликованную версию из legacy-настроек. Для `SHADOW` legacy-представление всегда неактивно: предыдущий backend не умеет отличать наблюдение от `LIVE` и после отката не должен начать менять статусы комментариев.

Presets имеют стабильные значения:

| Preset | Review | Spam | Чистый комментарий |
| --- | ---: | ---: | --- |
| `OPEN` | 70 | 130 | `APPROVE` |
| `BALANCED` | 45 | 90 | `APPROVE` |
| `STRICT` | 25 | 85 | `FOLLOW_SITE_MODE` |
| `CUSTOM` | явное значение | явное значение | явная настройка |

Решение `SPAM` всегда даёт итоговый статус `SPAM`, решение `REVIEW` — `PENDING`. Для `APPROVE` итог зависит от clean action и режима сайта. В режиме политики `SHADOW` решение и сигналы сохраняются, но публичный статус остаётся базовым.

| Режим сайта | Базовый статус | `SHADOW`: любое решение | `LIVE`: `APPROVE` + auto-approve | `LIVE`: `APPROVE` + follow-site | `LIVE`: `REVIEW` | `LIVE`: `SPAM` |
| --- | --- | --- | --- | --- | --- | --- |
| `PRE_MODERATION` | `PENDING` | `PENDING` | `APPROVED` | `PENDING` | `PENDING` | `SPAM` |
| `POST_MODERATION` | `APPROVED` | `APPROVED` | `APPROVED` | `APPROVED` | `PENDING` | `SPAM` |
| `DISABLED` | `APPROVED` | `APPROVED` | `APPROVED` | `APPROVED` | `PENDING` | `SPAM` |

`comments` получает snapshot последней оценки: ссылку на версию политики, режим применения, score, решение, безопасное объяснение `automod_reason`, время оценки, JSONB со стабильными кодами и весами сигналов, признак фактического применения и итоговый статус. Snapshot обновляется при редактировании вместе с повторной проверкой и живёт столько же, сколько комментарий. Он не содержит отдельной копии текста, совпавших фрагментов, email автора или URL страницы. `moderation_reason` остаётся совместимым полем применённого `LIVE`-решения; в `SHADOW` оно не заполняется и не влияет на приоритет старой очереди.

Миграция `V15__analytics_workload_events.sql` добавляет таблицу неизменяемых событий `automod_decision_events`. Триггер PostgreSQL после вставки комментария или изменения полей снимка автомодерации атомарно сохраняет ссылку на комментарий и версию политики, режим `SHADOW/LIVE`, решение, применённый статус и время оценки. Событие создаётся только для полного ненулевого снимка и только при фактическом изменении решения или времени оценки; обычное редактирование других полей и очистка снимка событие не создают. Запись выполняется на уровне базы данных, поэтому продолжает работать после отката приложения на совместимый предыдущий релиз и не дублируется новым сервером. Текст, причины, сигналы, автор и URL не копируются. Это сохраняет историческую аналитику после следующего редактирования, которое обновляет снимок в `comments`. Миграция создаёт одно начальное событие из каждого существующего снимка; более ранние перезаписанные оценки восстановить невозможно. События каскадно удаляются вместе с комментарием или сайтом.

`automod_policy_feedback` связывает комментарий, применённую версию и владельца с одним из типов `FALSE_POSITIVE` или `FALSE_NEGATIVE`. Текст комментария, автор, URL страницы и сырые сигналы в таблицу не копируются. Для одной пары comment/policy действует одно текущее значение; повторная отправка идемпотентна, а изменение оценки владельца заменяет тип без создания двойного вклада в аналитику. Детальная обратная связь хранится 90 дней и удаляется плановой retention-задачей либо раньше каскадно вместе с комментарием, сайтом или владельцем.

Моделирование политики использует тот же evaluator, что создание и редактирование комментария, но не создаёт комментарий, не меняет активную версию и не сохраняет переданный текст. Ответ содержит версию, правила, score, решение и безопасные объяснения. Чужие и отсутствующие сайты, версии и комментарии одинаково маскируются как `404 NOT_FOUND`.

### Умная очередь модерации

Очередь модерации не добавляет новых таблиц: `priority`, `priorityScore` и `priorityReasons` вычисляются при чтении из существующих полей `comments.status`, `comments.moderation_reason`, `comments.body`, `comments.parent_id` и `comments.created_at`. Сортировка по умолчанию `SMART DESC` поднимает выше комментарии, которые дольше ждут решения или имеют явные сигналы риска: автомодерация, ссылки/контакты, длинный текст, spam-статус и ответы внутри обсуждений. Компактный интерфейс по умолчанию объединяет `PENDING` и `SPAM`, хранит фильтры локально, поддерживает прямые ссылки на комментарий и отправляет массовые действия одним идемпотентным запросом.

### Realtime-уведомления

Миграция V12 добавляет `owner_notifications` для постоянных уведомлений владельца о новых комментариях. Запись содержит только `owner_id`, `comment_id`, безопасный ключ дедупликации, `read_at` и время создания. Название сайта, адрес страницы, автор, статус и фрагмент текста берутся при чтении по текущим связям `comments -> pages -> sites`; копии содержимого комментария, URL и иных потенциально чувствительных данных в уведомлении не сохраняются.

Уникальность `(owner_id, deduplication_key)` делает повторную обработку события идемпотентной. Все чтения и изменения признака прочтения ограничены владельцем; чужой или отсутствующий UUID одинаково маскируется как `404 NOT_FOUND`. Уведомления хранятся 90 дней и удаляются плановой retention-задачей, а каскадные внешние ключи очищают их вместе с владельцем или комментарием.

После фиксации комментария backend сначала создаёт или находит постоянное уведомление, затем отправляет активным WebSocket-сессиям владельца событие с тем же `notificationId`. После действий модерации backend отправляет оперативное событие по данным `comments` и `moderation_actions`. Короткоживущие одноразовые WebSocket ticket-ы хранятся только в памяти backend и атомарно удаляются при handshake. Очередь модерации и аналитические read-запросы остаются источниками истины для состояния комментариев.

### Диагностика установки виджета

Миграция V13 добавляет `site_widget_health` с одной строкой на сайт. Таблица хранит только последний успешный и последний отклонённый origin в нормализованном формате `scheme://host[:port]` и время каждого события. Пути, query-параметры, fragment, содержимое страниц и ответы клиента не сохраняются. Backend не загружает страницы владельца и не принимает адрес для серверного запроса, поэтому диагностика не создаёт SSRF-поверхность.

Успешный ответ публичного `/config` после domain-policy проверки обновляет `last_successful_*`, а отклонение фактического `GET /config` или его CORS preflight для метода `GET` и известного site ID — `last_rejected_*`. Другие HTTP-методы и public-маршруты диагностические события не создают. Для отсутствующего site ID запись не создаётся. Сведения об отклонениях очищаются через 30 дней; успешное время остаётся для определения `STALE`. Статус `HEALTHY` означает успех за последние 24 часа с origin, который всё ещё входит в текущую конфигурацию. Такой подтверждённый успех имеет приоритет над последующим публичным отклонением и защищает диагностику от поддельного `Origin`; более новая отклонённая попытка всё равно видна владельцу отдельным предупреждением. Без свежего успеха более новое отклонение даёт `REJECTED`; неактивный сайт также получает `REJECTED`, старый или удалённый из конфигурации успешный origin — `STALE`, отсутствие событий — `NEVER_SEEN`.

`SPAM`-комментарии остаются в базе для модерации и не видны в публичном списке виджета. Пользователь виджета получает только нейтральное сообщение "отправлено на проверку"; внутренние spam-сигналы показываются только владельцу в админской модерации.
