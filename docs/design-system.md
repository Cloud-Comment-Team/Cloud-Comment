# Дизайн-система CloudComment

## Назначение

Этот документ задаёт исполнимую визуальную и интерфейсную систему ближайшей версии CloudComment. Базовая продуктовая модель — self-service для одного владельца сайта: он самостоятельно подключает виджет, читает обсуждения, отвечает посетителям и разбирает проблемные комментарии. Команды, приглашения и отдельные рабочие места модератора не определяют текущую композицию.

Система называется **Editorial Thread**. Её метафора — редакционная полоса с живой нитью разговора: текст и контекст важнее контейнеров, статус ведёт к действию, а связь между репликами видна без тяжёлых вложенных карточек.

Документ обязателен для admin-панели, публичного виджета, Storybook, visual regression и новых UX-текстов. Он не разрешает косметически перекрашивать старую структуру: сначала выбирается подходящая композиция, затем применяются токены.

## Принципы

1. **Разговор раньше управления.** В обсуждении сначала видны автор, текст, контекст и продолжение ветки; служебные действия появляются рядом только по необходимости.
2. **Состояние ведёт к следующему шагу.** Ошибка, пустой результат и предупреждение всегда объясняют причину и предлагают одно основное действие.
3. **Одна поверхность — одна задача.** «Обсуждения» предназначены для чтения и ответа, «К разбору» — для решений модерации, «Сайты» — для подключения и настройки.
4. **Иерархия строится ритмом.** Типографика, интервалы, линии и плотность используются раньше border, background и shadow.
5. **Карточка означает самостоятельный объект.** Строки, сообщения, метрики и настройки не получают карточку автоматически.
6. **Тихая база, точный акцент.** Бирюзовый обозначает главное действие, focus и выбранное состояние; он не создаёт неоновое свечение.
7. **Публичный виджет принадлежит странице.** Он наследует шрифт сайта, не загружает собственные шрифты и не выглядит уменьшенной админкой.
8. **Доступность является состоянием компонента.** Focus, клавиатура, screen reader, reduced motion и контраст проектируются вместе с default и hover.
9. **Скорость — часть визуального качества.** Стабильный layout и сохранённый контекст важнее entrance-анимации.

## Визуальный язык Editorial Thread

Основная плоскость напоминает современную цифровую редакцию: тёплый нейтральный фон, чернильный текст, тонкие разделители и одна вертикальная нить обсуждения. Акцент появляется в местах принятия решения, а не на каждом интерактивном элементе.

- Заголовок страницы формулирует задачу, а не повторяет название раздела декоративным eyebrow.
- Временная последовательность читается сверху вниз; линии соединяют связанные события и ответы.
- Системная информация размещается в полях контекста и строках, а не в сетке одинаковых KPI-карточек.
- Активные обсуждения отличаются непрочитанным маркером и последней репликой, а не цветной рамкой всей строки.
- Предупреждение использует текст, знак и действие; цвет никогда не является единственным сигналом.

### Правила цвета

- Обычный текст и ссылки меньше 24 пикселей обеспечивают контраст не ниже `4.5:1`.
- Крупный текст, focus, границы контролов и значимые иконки обеспечивают не ниже `3:1`.
- Primary button использует `accent` как фон и контрастный текст, рассчитанный для конкретной темы; один и тот же foreground не переносится между темами автоматически.
- Warning и danger не заменяют подпись, иконку и доступное имя.
- Custom accent виджета проверяется на всех фактических парах background/foreground. Проверка только `3:1` для цвета в целом недостаточна.
- Axe `color-contrast` не отключается ни в одном обязательном E2E-профиле.

## Типографика

### Семейства

- Admin UI: `Inter, ui-sans-serif, system-ui, sans-serif` до отдельного измеренного решения о фирменном шрифте.
- Редакционный акцент допустим только для page title и hero-момента первого подключения: `Literata, Georgia, serif`. Шрифт подключается self-hosted Cyrillic/Latin subset и только если не ухудшает LCP; без него используется Georgia.
- Публичный widget: `font-family: inherit`. Виджет не загружает Inter, Literata или другой webfont и не сбрасывает шрифт host page.
- Числа в таблицах и метриках используют `font-variant-numeric: tabular-nums`.

### Шкала

| Роль | Размер / строка | Вес | Где использовать |
|---|---:|---:|---|
| Display | `40 / 44` | `600` | один hero-момент onboarding |
| Page title | `32 / 38` | `650` | заголовок рабочего экрана |
| Section title | `22 / 28` | `650` | крупный раздел страницы |
| Thread title | `18 / 26` | `650` | заголовок ветки и detail |
| Body | `16 / 25` | `400` | текст комментария, описание |
| UI body | `14 / 21` | `450` | controls, строки, metadata |
| Label | `13 / 18` | `600` | label поля и статус |
| Caption | `12 / 17` | `500` | время и вторичный контекст |

Page title не сопровождается постоянным uppercase eyebrow. Eyebrow допустим только для настоящего контекста над заголовком, например имени сайта над названием обсуждения.

## Пространство и плотность

Базовый модуль равен 4 пикселям. Разрешённые интервалы:

| Токен | Значение | Типичное применение |
|---|---:|---|
| `--space-0` | `0` | сброс |
| `--space-0-5` | `2px` | оптическая коррекция |
| `--space-1` | `4px` | связь label и caption |
| `--space-2` | `8px` | icon gap, compact row |
| `--space-3` | `12px` | control group |
| `--space-4` | `16px` | строка, mobile gutter |
| `--space-6` | `24px` | секция, desktop gutter |
| `--space-8` | `32px` | крупный блок |
| `--space-12` | `48px` | разделы страницы |
| `--space-16` | `64px` | редкий hero interval |

Запрещены произвольные отступы, если значение отличается от шкалы менее чем на 3 пикселя и не является вычисляемым safe-area offset. Compact mode уменьшает вертикальный ритм строк и сообщений, но не уменьшает touch targets, focus ring и читаемость текста.

## Сетка и ширина

- Desktop от 1280 пикселей: навигация 224 пикселя, content gutter 32 пикселя, рабочая область до 1280 пикселей, 12 колонок с gutter 24 пикселя.
- 1024–1279: навигация 200 пикселей, content gutter 24 пикселя, 8 колонок.
- 768–1023: компактная навигация 72 пикселя либо явная drawer-навигация; 8 колонок, gutter 20 пикселей.
- До 767: одна колонка, gutter 16 пикселей, верхняя app bar и menu drawer. Боковая панель не остаётся второй конкурирующей навигацией.
- Текстовая колонка discussion detail ограничивается 760 пикселями.
- Формы настройки ограничиваются 680 пикселями, если справа нет реального preview/context.
- Таблица занимает доступную ширину; описательный текст внутри таблицы не растягивает колонку без ограничения.

## Радиусы и elevation

| Токен | Значение | Применение |
|---|---:|---|
| `--radius-none` | `0` | строки, таблицы, thread rail |
| `--radius-xs` | `3px` | небольшой status marker |
| `--radius-sm` | `6px` | input, compact button |
| `--radius-md` | `10px` | самостоятельная panel/empty state |
| `--radius-lg` | `14px` | modal, mobile sheet |
| `--radius-round` | `999px` | avatar и настоящий badge, не обычная кнопка |

| Уровень | Реализация | Применение |
|---|---|---|
| `flat` | без тени; разделитель или смена фона | list, table, thread |
| `outlined` | `1px border` без тени | input, самостоятельная panel |
| `raised` | мягкая тень `0 8px 24px rgba(16,24,20,.10)` | sticky action bar, popover |
| `overlay` | тень `0 18px 48px rgba(8,12,10,.22)` | modal/drawer |

Компонент не сочетает заметную границу, отдельный цвет фона и тень одновременно без письменного обоснования. Комментарии и строки таблицы используют `flat`, не `raised`.

## Иконки и графика

- Иконки имеют размер 16, 18, 20 или 24 пикселя и толщину линии, согласованную в одном наборе.
- Иконка без текста допустима только для общепринятого действия и всегда имеет accessible name и tooltip при неочевидности.
- Иллюстрации используются в первом подключении и редких пустых состояниях. Их мотив — нить, реплики и редакционные пометки, а не 3D-сферы, роботы или абстрактный AI glow.
- Декоративные изображения не заменяют инструкцию, статус и действие.

## Интерактивные состояния

### Controls

| Состояние | Требование |
|---|---|
| Default | доступное имя, предсказуемая форма, достаточный контраст |
| Hover | изменение background/border без перемещения элемента и изменения layout |
| Focus visible | ring 2 пикселя `focus`, offset 2 пикселя; не обрезается контейнером |
| Pressed/selected | видимый текстовый или графический признак и `aria-pressed`/`aria-selected` |
| Disabled | причина доступна рядом; opacity не является единственным сигналом |
| Busy | действие сохраняет ширину, показывает progress и блокирует повтор только своей операции |
| Error | ошибка связана с полем через `aria-describedby`; введённое значение сохраняется |
| Success | подтверждается рядом с действием или объектом, не отдельным блокирующим modal |

### Loading, empty, error и stale

- Loading до 200 миллисекунд не показывает skeleton. После порога появляется каркас фиксированной высоты.
- Refetch сохраняет уже загруженные данные видимыми, если они не стали недействительными; область помечается `aria-busy`.
- Empty state содержит причину, следующий шаг и одно основное действие. Текст «Здесь пока ничего нет» без контекста запрещён.
- Error не очищает draft, выбранные безопасные фильтры и предыдущие данные. Retry повторяет только сломанный запрос.
- Stale state показывает время последнего успешного обновления и действие «Обновить».
- Success и error объявляются через `role=status` или `role=alert` по срочности.

## Базовые композиции

### Исполнимый контракт оболочки

Этот раздел проверяется на реальных маршрутах, а не только в Storybook.

- На ширине от 1024 пикселей верхняя app bar не рендерится: основная навигация, уведомления, профиль, выход и тема находятся в одной боковой панели шириной 224 пикселя.
- На ширине до 1023 пикселей боковая панель становится drawer, а единственная верхняя app bar содержит меню, бренд, уведомления и тему.
- На каждом viewport существует ровно по одному переключателю темы и кнопке уведомлений. Скрытая копия элемента в DOM не допускается.
- Заголовок страницы не помещается в отдельную карточку. Он задаёт контекст и одно основное действие, после него начинается рабочая поверхность.
- Eyebrow над заголовком допускается только если сообщает новый контекст. На маршруте `/sites` подпись «Проекты» запрещена как декоративный дубль заголовка «Сайты».
- Списки используют одну адаптивную разметку: на desktop это плотные строки с колонками, на mobile те же строки перестраиваются без второй версии DOM.
- На 1440 × 900 список сайтов показывает не менее семи строк без вертикальной прокрутки при стандартном наборе данных.
- Опасные действия визуально и семантически отделены от повседневных настроек, но не превращаются в полноэкранную предупреждающую карточку.

Обязательные маршруты первой проверки: `/sites` и `/account`. Для каждого сохраняются эталоны 390 × 844, 768 × 1024 и 1440 × 900 в светлой/тёмной теме и проверяется `prefers-reduced-motion`.

## Lists, tables и master-detail

### Lists

Список используется, когда объекты читаются последовательно и главное различие находится в тексте: обсуждения, уведомления, сайты с одной основной проблемой.

- Строка имеет один primary target на всю безопасную область.
- Первые две строки текста содержат решение: название/фрагмент и статус/контекст.
- Дополнительные действия появляются при focus within, hover или через постоянное меню на touch.
- Unread обозначается маркером, шрифтом и accessible label.
- Разделитель проходит между строками; каждая строка не получает rounded card.

### Tables

Таблица используется для сопоставления нескольких одинаковых полей и массового выбора: очередь модерации, версии политик, техническая история.

- Заголовок остаётся видимым при вертикальной прокрутке длинной очереди.
- Чекбокс выбирает только видимые элементы и явно сообщает количество.
- Числа выравниваются по правому краю, текст — по левому.
- Status не занимает отдельную цветную карточку.
- На узком экране таблица либо сохраняет горизонтальный scroll с закреплённым главным столбцом, либо превращается в специально спроектированные строки. Скрывать смысловые столбцы без альтернативы запрещено.

### Master-detail

Master-detail используется в «Обсуждениях» и «К разбору», но с разными наборами действий.

- Desktop: master 320–380 пикселей, detail занимает остаток; выбранная строка и detail имеют общий URL.
- Tablet: master 40%, detail 60%, если detail остаётся не уже 520 пикселей; иначе отдельные экраны.
- Mobile: список и detail — разные адресуемые маршруты; Back восстанавливает scroll, filters и selection.
- Detail drawer применяется только для краткой вспомогательной информации. Основной разговор и длинная форма не помещаются в modal drawer на desktop.
- Любой modal drawer использует portal, backdrop, `aria-modal`, focus trap, Escape, return focus и внутренний `overflow-y:auto`.

## Обсуждения

Discussion row показывает сайт, страницу, безопасное имя последнего автора, последнюю реплику, время, число ответов и состояние «ждёт ответа». Приоритет вычисляется из непрочитанности и ожидания ответа, а не из automod score.

Thread detail:

- сохраняет ширину текста 52–78 символов;
- показывает root и ответы одной нитью;
- отличает ответ владельца текстовой меткой и мягким accent background только в metadata;
- не повторяет четыре пустые reaction-кнопки под каждой короткой репликой;
- предоставляет permalink только после отдельного продуктового решения, не имитирует его неработающей кнопкой;
- сохраняет owner reply draft при error, route change и refetch;
- после успешного owner reply снимает «ждёт ответа» без перезагрузки всего master-list.

## К разбору

Moderation surface оптимизирована под одно решение за несколько секунд.

- Master показывает текст, безопасного автора, статус, причину и возраст ожидания.
- Detail сначала показывает контекст комментария, затем рекомендацию и только потом технические сигналы.
- Primary action соответствует наиболее вероятному безопасному решению, остальные остаются secondary.
- Keyboard shortcuts работают только вне inputs и всегда видимы в подсказке.
- Bulk action bar закрепляется у края viewport, показывает количество и точное последствие.
- Undo после bulk относится ко всей operation. При частичном результате перечисляется количество отменённых и неотменённых элементов.
- Drawer не используется как единственный способ добраться до нижних действий на mobile; mobile detail является отдельным экраном.

## Онбординг первого сайта

Критический путь имеет строго три продуктовых этапа:

1. **Сайт.** Владелец вводит название, адрес/origin и выбирает понятный режим публикации. Система применяет безопасные defaults. Детальная автомодерация, оформление и аналитика не входят в этот шаг.
2. **Установка.** После создания владелец сразу получает реальный snippet, разрешённый origin и проверку фактического запроса widget. Успех определяется `HEALTHY`, а не нажатием «Готово».
3. **Первый реальный комментарий.** Владелец открывает настоящую внешнюю страницу, публикует комментарий через production widget и видит его в CloudComment. Синтетический комментарий из панели, seed и preview не завершают онбординг.

Последовательность не меняется на «сайт → оформление → правила → установка». Оформление и advanced policy предлагаются после первого реального комментария как необязательная настройка.

Progress показывает только подтверждённые события. Если origin отклонён, этап установки остаётся активным, показывает нормализованный origin и одно действие исправления. Draft создания сайта сохраняется локально owner-scoped; retry не очищает поля.

## Публичный widget

### Структура

1. Компактный header: название, количество, сортировка.
2. Поток корневых комментариев с постраничной загрузкой.
3. Collapsed replies и lazy loading без дублей.
4. Composer, который можно начать заполнять до авторизации.
5. Компактная account affordance вне основного потока.

### Состояния комментария

- `sending` — локальная строка с текстом и progress; не маркируется преждевременно как pending.
- `approved` — обычная опубликованная реплика.
- `pending` — видна только автору, содержит «На проверке» и время отправки.
- `failed` — сохраняет текст и даёт «Повторить»/«Редактировать».
- `deleted` — tombstone сохраняет структуру ветки и ответы.
- `edited` — спокойная подпись «изменено»; полная история не обещается, пока её нет в продукте.

### Действия

- Primary action комментария — «Ответить».
- Reaction summary показывает существующие реакции; добавление открывается одной компактной affordance.
- Edit/delete собственного сообщения находятся в context menu и доступны клавиатурой.
- Delete root с ответами сохраняет tombstone либо заранее честно сообщает точное последствие.
- Авторизация запоминает исходное намерение: comment, reply или reaction. После успеха действие продолжается, а draft не теряется.

### Интеграция с сайтом

- Widget наследует font family, line height и доступный color scheme host page, если owner не выбрал явную тему.
- Shadow DOM изолирует внутреннюю раскладку, но не считается изоляцией bearer token.
- Container queries являются главным механизмом responsive; viewport media query используется только как fallback.
- Default page identity удаляет fragment и следует canonical URL; integrator может передать стабильный `pageKey/pageUrl` явно.
- Виджет не загружает шрифты, иллюстрации, analytics SDK и motion library.

## Адаптивность

Обязательные ширины проверки: 320, 390, 768, 1024, 1280 и 1440 пикселей.

| Диапазон | Admin | Widget |
|---|---|---|
| 320–389 | одна колонка; full-screen detail; sticky primary action с safe area | один поток; sort на отдельной строке; composer на всю ширину |
| 390–767 | одна колонка; menu drawer; filters в раскрываемой области | metadata переносится, thread rail сохраняется, targets не меньше 44 пикселей |
| 768–1023 | компактная навигация; master/detail только при detail от 520 пикселей | container layout, не viewport layout |
| 1024–1279 | двухпанельные рабочие поверхности, gutter 24 | readable width по настройке владельца |
| 1280+ | постоянная навигация, master/detail, content max 1280 | max width не растягивает строку текста сверх читаемой |

Layout не должен создавать горизонтальный scroll страницы. Допустим локальный scroll таблицы с видимой подсказкой. Virtual keyboard не перекрывает composer и primary action.

## Доступность

- Цель — WCAG 2.2 AA без отключённых critical/serious axe rules.
- Landmark, heading hierarchy и accessible name отражают пользовательскую задачу.
- Tabs используют tablist/tab/tabpanel; toggle — `aria-expanded` и `aria-controls`; reaction — `aria-pressed`.
- Dynamic status объявляется через live region без повторного чтения всей страницы.
- Focus не теряется при optimistic update, refetch и закрытии overlay.
- Touch target на mobile не меньше 44 на 44 пикселя; минимальная интерактивная область desktop — 28 пикселей при достаточном расстоянии.
- Интерфейс работает при zoom 200%, text-only zoom и системном forced-colors.
- Reduced motion сохраняет причинно-следственную связь мгновенной сменой состояния; transform-hover и entrance-анимации также отключаются.
- Error summary ведёт к первому неверному полю; label и инструкция не зависят от placeholder.
- Public email отсутствует в DOM, accessible name и скрытом тексте.

## Производительность

- Widget bundle: не более 20 КБ gzip; новые зависимости требуют отдельного обоснования.
- Основной admin bundle: не более 180 КБ gzip; обычный route bundle — не более 60 КБ gzip согласно действующему quality budget.
- Widget не выполняет полный `replaceChildren` при локальном действии и не переанимирует весь список.
- Корневые комментарии загружаются страницами. Replies загружаются по запросу; DOM не содержит скрытые сотни сообщений.
- Skeleton имеет итоговые размеры и не создаёт layout shift.
- Изображения вне первого viewport lazy-load; decorative media не входит в critical path.
- Анимации используют transform/opacity, не блокируют input и не запускаются для всех строк одновременно.
- Performance-тесты измеряют настоящий API page size, а не mock, возвращающий сто элементов вопреки запросу двадцати.

## Разрешено

- Тёплые нейтральные плоскости и высокий текстовый контраст.
- Тонкая thread rail для связи ответов.
- Один accent на primary action, focus и выбранном объекте.
- Split view для разговоров и очереди на широком экране.
- Inline status, local retry и optimistic update с сохранённым draft.
- Card для самостоятельного onboarding-блока, modal или независимого site object.
- Serif-акцент в редком editorial heading при соблюдении performance budget.
- Контекстные действия, доступные одновременно мышью, touch и клавиатурой.

## Запрещено

- Neon glow, glassmorphism и cyan border вокруг всей рабочей области.
- Одинаковые rounded cards для каждого текста, ответа, числа и setting group.
- Gradient text, 3D-сферы и AI-иллюстрации без пользовательской функции.
- Цветные pills как универсальная навигация или кнопка.
- Постоянные четыре reaction button под каждым сообщением с нулевыми значениями.
- Hover-only действия без keyboard/touch эквивалента.
- Skeleton при каждом быстром refetch и исчезновение уже прочитанного списка.
- Modal для частого решения, которое должно выполняться inline или в detail.
- Дублирующая desktop-навигация и два переключателя темы.
- Фальшивое завершение установки по нажатию кнопки без реального запроса widget.
- Fake data и seed в production-like интерфейсе вне явно названного demo/Storybook.
- Отключение axe rule ради зелёного CI.

## Миграция без big bang rewrite

### Этап 0. Контракты и безопасность

- убрать публичный email;
- стабилизировать page identity;
- изолировать widget/admin session;
- сохранить draft/focus и pending lifecycle;
- вернуть обязательный color-contrast test.

Визуальный редизайн не должен маскировать эти дефекты.

### Этап 1. Токены и primitives

- ввести role-based theme tokens;
- обновить Button, Field, Status, ListRow, Table и Overlay;
- сделать Drawer доступным и прокручиваемым;
- запретить новые raw colors и произвольные radii lint/review-правилом.

Legacy и новые primitives не смешиваются внутри одного экрана.

### Этап 2. Первый вертикальный сценарий

Перенести только завершённый путь **сайт → установка → первый реальный комментарий**. Он становится эталоном spacing, states, writing, focus и responsive.

### Этап 3. Today и Discussions

- заменить dashboard операционным Today;
- добавить owner discussion API и master-detail;
- обеспечить owner reply, unread и сохранение позиции.

Подробная аналитика остаётся отдельной страницей.

### Этап 4. К разбору

- перейти на list/detail composition;
- исправить bulk undo;
- заменить текущий недоступный Drawer;
- закрепить action bar и keyboard flow.

### Этап 5. Публичный widget

- внедрить Editorial Thread;
- добавить root pagination, collapsed replies и persistent composer;
- обновить auth intent и состояния комментария;
- проверить реальные host fonts, narrow containers и обе темы.

### Этап 6. Удаление legacy

- удалить старые cards и неиспользуемые style overrides;
- убрать фиктивные настройки и technical demo text;
- обновить Storybook, screenshots, docs и E2E;
- удалить временные adapters только после production smoke и одного совместимого релиза.

## Definition of Done

- На экране нет параллельных старой и новой дизайн-систем.
- Один владелец без внешней помощи проходит сайт → установка → первый реальный комментарий.
- Today отвечает, что сломано, что изменилось и что сделать первым.
- Discussions показывают активные разговоры и owner reply, а не очередь модерации.
- «К разбору» выполняет одиночное и массовое решение с честным undo.
- Widget не раскрывает email, не теряет draft/focus и предоставляет все корневые комментарии страницами.
- Drawer, dialog, tabs, reactions и dynamic status доступны клавиатуре и screen reader.
- Color contrast проверяется без исключений; critical/serious axe violations отсутствуют.
- Проверены 320, 390, 768, 1024, 1280 и 1440 пикселей, light/dark и reduced motion.
- Соблюдены bundle budgets, нет заметного layout shift и массового повторного render.
