# Интерфейс CloudComment

Этот файл содержит только устойчивые правила интерфейса. Планы, аудиты и требования к отдельным задачам хранятся в GitHub Issues; актуальные состояния компонентов — в Storybook и visual-тестах.

## Основные принципы

- Текст, контекст и следующее действие важнее декоративных контейнеров.
- Карточка используется для самостоятельного объекта, а не для каждой строки или группы текста.
- Бирюзовый акцент обозначает основное действие, выбранное состояние и фокус. Свечение и декоративные градиенты не используются.
- Hover не перемещает элементы. Большие поверхности не масштабируются и не размываются.
- Ошибка, загрузка и пустое состояние сохраняют контекст и предлагают понятный следующий шаг.

## Оболочка

- От 1024 пикселей используется одна боковая панель шириной 224 пикселя. Отдельной верхней панели нет.
- До 1023 пикселей боковая панель становится drawer, а мобильная шапка содержит меню, бренд, уведомления и тему.
- На каждом viewport в DOM существует ровно одна кнопка темы и одна кнопка уведомлений. Скрытые дубликаты запрещены.
- Заголовок страницы не помещается в карточку. Eyebrow допустим только для нового контекста; например, подпись «Проекты» над заголовком «Сайты» запрещена как повтор.

## Цвет и поверхности

Компоненты используют семантические CSS-переменные из `frontend/admin/src/index.css`: фон, поверхность, текст, границу, акцент и состояния success/warning/danger. Raw hex внутри компонентов не добавляется без отдельного обоснования.

- Светлая тема использует тёплый нейтральный фон и чернильный текст.
- Тёмная тема использует спокойные зелёно-серые поверхности без неонового свечения.
- Обычный текст соответствует WCAG AA `4.5:1`; значимые границы, focus и крупный текст — не ниже `3:1`.
- Цвет не является единственным признаком статуса.

## Типографика и плотность

- Admin UI использует системный sans-serif; виджет наследует шрифт страницы и не загружает собственный webfont.
- Базовые роли: заголовок страницы 32 пикселя, заголовок секции 16–22, основной текст 16, интерфейсный текст 14, подпись 12–13.
- Отступы кратны 4 пикселям. Основные значения: 8, 12, 16, 24, 32 и 48.
- Touch target не уменьшается ниже 40 пикселей даже в compact-режиме.
- На 1440 × 900 рабочий список должен показывать не менее семи строк без вертикальной прокрутки.

## Анимация

- 120 мс: цвет, граница, небольшой индикатор.
- 180 мс: локальное появление строки или сообщения.
- 240 мс: drawer или связанный переход.
- Spring допустим только для небольшого индикатора. Bounce, blur и scale больших поверхностей запрещены.
- `prefers-reduced-motion` отключает декоративное движение без потери состояния и фокуса.

## Адаптивные списки и формы

- Список использует одну DOM-структуру: desktop-колонки перестраиваются в мобильную строку без второй версии данных.
- Формы ограничивают ширину, если рядом нет реального preview или контекста.
- Опасные действия отделяются от повседневных настроек и требуют явного текста последствия.
- Публичный виджет использует container queries и не зависит от ширины viewport хост-страницы.

## Композиционные эталоны маршрутов

- `/` («Сегодня»): не более трёх приоритетных поверхностей; установка, очередь и обсуждения идут раньше любых метрик. Нулевые KPI и графики находятся только в `/analytics`.
- `/sites/:siteId?section=appearance`: controls и реальный preview видны одновременно на desktop; на узких экранах URL секции сохраняет контекст после reload/back. Save-панель не перемещается, опасная зона является отдельной секцией.
- `/comments`: compact master/detail; строки разговора не являются тяжёлыми карточками, composer остаётся частью detail и сохраняет draft.
- `/moderation`: compact очередь и контекст решения; bulk-панель появляется только после выбора, история отделена от `PENDING`/`SPAM`.
- Публичный виджет: корень и ответы образуют одну ветку без внешней панели и вложенных карточек; composer является главным действием, auth раскрывается по намерению пользователя.

## Проверяемые motion-storyboard

1. Переход между соседними маршрутами: 180 мс, не более 8 пикселей; уже доступные данные не заменяются пустым экраном.
2. Открытие обсуждения: меняется только detail, список и shell остаются неподвижными; фокус переходит в заголовок.
3. Отправка ответа: composer сохраняет высоту, optimistic/error/retry заменяют состояние без перестройки страницы.
4. Bulk-модерация и undo: панель появляется в зарезервированном контексте, результат операции немедленно прерывает устаревшее движение.
5. Уведомления: триггер не меняет положение, закрытие возвращает фокус; mobile использует полноэкранную поверхность.

В reduced-motion те же storyboard выполняются без пространственного движения; live-region, focus, optimistic state и undo остаются доступными.

## Доступность и проверка

- Все интерактивные элементы имеют доступное имя и видимый `focus-visible`.
- Dialog и drawer удерживают фокус, закрываются по Escape и возвращают фокус триггеру.
- В обязательной матрице проверяются 390 × 844, 768 × 1024 и 1440 × 900, светлая/тёмная тема и reduced motion.
- Источник истины для внешнего вида — Storybook и снимки в `frontend/admin/e2e/ux/__screenshots__`.
- Изменение интерфейса принимается только после visual regression, axe, unit-тестов и production smoke; зелёная сборка без визуальной проверки недостаточна.
